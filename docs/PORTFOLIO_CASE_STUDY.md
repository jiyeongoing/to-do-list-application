# Swipe Todo - Project Case Study

## 1. Project Summary

`Swipe Todo`는 iPhone 사용자를 위한 모바일 PWA 투두리스트 프로토타입입니다.
오늘 실행할 일은 단순하게 보고, 미래 할 일과 장보기/여행 준비 같은 목적별
리스트는 날짜에 미리 작성해 해당 날짜에 확인하는 흐름을 검증했습니다.

| 항목 | 내용 |
| --- | --- |
| Role | Product Planner / Prototype Builder |
| Stage | Usable PWA Prototype |
| Platform | iPhone-first PWA Web |
| Storage | LocalStorage Prototype |
| Demo | https://jiyeongoing.github.io/to-do-list-application/?v=12 |
| Repository | to-do-list-application |

## 2. Problem

기존 투두 앱은 오늘 할 일, 반복 루틴, 미래 계획, 목적별 체크리스트가 한 화면에
섞이기 쉽습니다. 사용자는 지금 해야 할 일에 집중하고 싶지만, 동시에 다음 주
일정이나 여행 준비처럼 미래에 필요한 목록도 미리 적어두고 싶어합니다.

핵심 문제는 다음과 같이 정의했습니다.

- 오늘 화면은 복잡하지 않아야 한다.
- 반복 루틴은 매일 다시 입력하지 않아야 한다.
- 미래에 적어둔 할 일은 해당 날짜에 자동으로 보여야 한다.
- 장보기/여행 준비처럼 묶음 목록은 별도 화면에서 집중해서 봐야 한다.

## 3. Product Direction

초기 구조는 `오늘 / 이번주 / 데일리`였지만, 사용 흐름을 줄이기 위해
`오늘 / 데일리 / 계획`으로 정리했습니다. 기본 화면은 오늘과 데일리만 두고,
계획은 스와이프 또는 버튼으로 이동하는 보조 화면으로 설계했습니다.
현재 버전에서는 오늘 화면에서 오른쪽으로 스와이프하면 계획으로 이동하고,
계획 화면에서 왼쪽으로 스와이프하면 오늘로 돌아옵니다.

| 화면 | 역할 |
| --- | --- |
| 오늘 | 지금 처리할 단일 할 일과 오늘 도착한 리스트 확인 |
| 데일리 | 매일 반복될 루틴 원본 관리 |
| 계획 | 날짜별 할 일과 목적별 리스트 사전 작성 |
| 리스트 상세 | 장보기/여행 준비 같은 묶음 항목 체크 |

## 4. Key UX Decisions

### 오늘 리스트 위치

오늘 화면에서는 단일 투두를 먼저 보여주고, 그 아래에 `오늘 리스트` 카드를
배치했습니다. 단일 투두는 즉시 실행성이 높고, 리스트는 묶음 작업이라 카드로
한 단계 낮게 두는 편이 사용자의 시선 흐름에 더 자연스럽다고 판단했습니다.

### 과거 날짜 정책

오늘보다 과거인 날짜는 다른 색으로 표시하고 읽기전용으로 처리했습니다. 이미
지난 계획을 수정하기보다 기록처럼 확인하는 성격이 강하기 때문입니다.

### 리스트 복사

장보기나 여행 준비처럼 반복되는 목적별 리스트는 복사 후 원하는 날짜에
붙여넣을 수 있게 했습니다. 복사된 항목의 완료 상태는 새 날짜에서 다시 시작할
수 있도록 모두 미완료로 초기화했습니다.

### 버튼과 순서 변경

순서 변경은 리스트 우측의 작은 `▲/▼` 버튼으로 통일했습니다. 맨 위 항목은
아래 방향만, 맨 아래 항목은 위 방향만 보여 불필요한 버튼을 줄였습니다.

### 데일리 기본값

새로 추가한 데일리 루틴은 기본 `미사용` 상태로 둡니다. 사용자가 루틴을
만든 즉시 오늘 목록에 섞이는 일을 막고, 실제로 반복할 항목만 직접 켜도록
했습니다.

## 5. Prototype Scope

현재 프로토타입에서 구현한 범위는 다음과 같습니다.

- 오늘 할 일 추가, 완료, 삭제, 순서 변경
- 오늘 할 일 날짜별 저장 및 해당 날짜 표시
- 데일리 루틴 추가, 기본 미사용, 사용/미사용, 삭제, 순서 변경
- 계획 진입 시 오늘 날짜 자동 선택
- 계획 날짜 이동, 오늘 이동, 달력 이동
- 날짜별 일반 할 일 추가, 삭제, 순서 변경
- 목적별 리스트 생성, 삭제, 항목 추가, 완료, 순서 변경
- 오늘 도착 리스트 카드와 완료 수 재계산
- 계획에서 오늘 날짜 선택 시 오늘 할 일 동기화
- 오늘 오른쪽 스와이프, 계획 왼쪽 스와이프 이동
- 과거 날짜 읽기전용 처리
- 리스트 복사/붙여넣기
- LocalStorage 기반 데이터 유지
- JSON 백업/복원과 데이터 비우기
- PWA manifest와 service worker 기반 앱 셸 캐시
- 게스트 저장 유지, Google 로그인 프로토타입, 로컬 데이터 계정 가져오기

## 6. Test Strategy

단순 화면 구현이 아니라 동작 안정성까지 고려했다는 점을 보여주기 위해
Node 기본 테스트 러너로 TDD 성격의 테스트를 추가했습니다.

테스트는 외부 라이브러리 없이 작은 가짜 DOM을 구성해 `app.js`의 실제 로직을
검증합니다.

```bash
node --test tests/todo.test.cjs
```

검증 케이스:

- 오늘 할 일 순서 변경
- 첫 실행 빈 목록
- 샘플 데이터 불러오기와 데이터 비우기
- 데일리와 계획 항목 순서 변경
- 새 데일리 루틴 기본 미사용
- 새 리스트명 유지 버그
- 오늘 리스트 완료 수 재계산
- 계획에서 오늘 날짜 선택 시 오늘 할 일 표시
- 날짜별 할 일 분리 표시
- 새 날짜의 활성 데일리 루틴 미완료 생성
- 게스트 저장과 계정 저장 전환
- 로컬 데이터 계정 가져오기
- 오늘/계획 스와이프 이동
- 계획 날짜 영역 스와이프 시 화면 전환 방지
- 계획에서 오늘 날짜에 추가한 항목의 오늘 저장
- 캘린더와 오늘 버튼 날짜 이동
- 계획하기 진입 시 오늘 날짜 초기화
- 과거 날짜 읽기전용 처리
- 리스트 복사/붙여넣기
- 맨 위/맨 아래 순서 버튼 표시 정책

## 7. Technical Notes

이번 단계는 Java/Spring Boot 서버 앱이 아니라, 제품 흐름 검증을 위한 정적
프론트엔드 프로토타입입니다.

| 영역 | 현재 | 다음 단계 |
| --- | --- | --- |
| UI | HTML/CSS/JavaScript | Spring Boot + Thymeleaf 또는 REST API |
| Storage | LocalStorage | IndexedDB, 이후 서버 DB |
| PWA | Manifest + Service Worker | iPhone 홈 화면 설치 테스트 |
| Test | Node test runner, 22 cases | 도메인 로직 분리 후 단위 테스트 확대 |
| Backend | - | Java/Spring Boot API 설계 |
| Deploy | GitHub Pages | iPhone 홈 화면 설치 검증 |

## 8. What I Learned

- 투두 앱은 기능 수보다 화면 분류가 중요합니다.
- 미래 계획은 오늘 화면과 분리해야 현재 실행성이 높아집니다.
- 목적별 리스트는 단일 투두와 다른 정보 구조로 다루는 편이 좋습니다.
- 프로토타입 단계에서도 테스트를 작성하면 잦은 UI 변경 중 핵심 동작을 지킬 수 있습니다.
- PWA는 네이티브 위젯은 어렵지만, 홈 화면 설치와 오프라인 재실행으로 앱 같은 진입 경험을 만들 수 있습니다.
